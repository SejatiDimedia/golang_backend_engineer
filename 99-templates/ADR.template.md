## Context

What problem or fork in the road prompted this decision. Include enough background that a reader with no other context understands *why a decision was needed at all*.

## Decision

State the decision in one or two sentences, unambiguously. This is the part someone skimming will read — make it stand alone.

## Alternatives considered

| Option | Pros | Cons |
|---|---|---|
| Option A (chosen) | | |
| Option B | | |

## Reasoning

Why the chosen option won, specifically in the context of this project's constraints (learning goals, time, scope) — not a generic "best practices" justification. If the honest reason is partly "this is the learning objective for this project," say so.

## Consequences

What this decision makes easier, and what it makes harder or forecloses. Every decision has both — if this section only lists upsides, it hasn't been thought through.

## Revisit conditions

What would change for this decision to be reconsidered (e.g. "if this service needs to scale past X writes/sec" or "once Project 7's Auth Service exists, this should be migrated"). This is what makes an ADR a living record instead of a one-time justification.
